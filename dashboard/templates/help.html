Help & Documentation

Learn about worker roles and how to use the multi-agent system

{# Quick Start Section #}

Quick Start

1 Create Work

Use the Dashboard to create beads (tasks, bugs, features) for agents to work on.

2 Spawn Workers

Go to Admin and spawn workers based on the type of work needed.

3 Monitor Progress

Watch agents work in real-time via Agents and Logs pages.

{# Worker Roles Section #}

Worker Roles

Each worker role has specific responsibilities and picks up beads with matching labels.

{# Developer Role #}

Developer

Label: dev
Most Common

What They Do

  • Write production code and implement features
  • Fix bugs and resolve issues
  • Create pull requests for changes
  • Refactor and improve code quality

Example Tasks

  • Implement new API endpoint
  • Add form validation logic
  • Fix null pointer exception
  • Optimize database queries

When to Spawn

When you have coding tasks, bug fixes, or features that need implementation.

{# QA Role #}

QA (Quality Assurance)

Label: qa

What They Do

  • Run test suites and verify functionality
  • Validate acceptance criteria
  • Create bug beads when issues found
  • Block feature beads on discovered bugs

Example Tasks

  • Verify new login feature works
  • Run regression test suite
  • Test edge cases for form inputs
  • Validate API responses

When to Spawn

After features are implemented to verify quality, or when you need thorough testing of existing functionality.

Note: QA workers do not write production code. They focus exclusively on testing and validation.

{# Tech Lead Role #}

Tech Lead

Label: architecture

What They Do

  • Review designs and architecture decisions
  • Create implementation task breakdowns
  • Set dependencies between tasks
  • Unblock complex technical decisions

Example Tasks

  • Design database schema for new feature
  • Break epic into implementable tasks
  • Decide on API structure
  • Resolve technical blockers

When to Spawn

For architecture decisions, breaking down complex features, or when technical guidance is needed.

{# Manager Role #}

Manager

Label: All beads (no filter)

What They Do

  • Create epics and high-level work items
  • Set priorities (P0-P4)
  • Assign labels and organize work
  • Generate status reports

Example Tasks

  • Create Q1 feature epic
  • Re-prioritize backlog items
  • Triage new bug reports
  • Review project health metrics

When to Spawn

For project planning, prioritization, or when you need to organize and structure work.

Note: Managers see all beads regardless of labels and focus on coordination, not implementation.

{# Reviewer Role #}

Code Reviewer

Label: review

What They Do

  • Review PR diffs for quality
  • Check code quality and standards
  • Verify tests exist and are adequate
  • Approve or request changes on PRs

Example Tasks

  • Review PR #123 for security issues
  • Check test coverage on new feature
  • Verify coding standards compliance
  • Review refactoring changes

When to Spawn

When PRs are ready for review, or when you need a second pair of eyes on code changes.

{# Priority Levels Section #}

Priority Levels

Workers pick up beads in priority order (P0 first, then P1, etc.)

P0

Critical

Drop everything

P1

High

Do next

P2

Medium

Default priority

P3

Low

When available

P4

Backlog

Nice to have

{# Workflow Tips Section #}

Workflow Tips

Best Practices

  • Label beads appropriately so the right workers pick them up
  • Use dependencies to ensure work is done in correct order
  • Set realistic priorities - not everything is P0
  • Write clear acceptance criteria in bead descriptions

Common Patterns

  • 1. Tech Lead breaks epic into tasks
  • 2. Developer implements features
  • 3. Reviewer reviews PRs
  • 4. QA validates functionality
  • 5. Manager tracks progress
